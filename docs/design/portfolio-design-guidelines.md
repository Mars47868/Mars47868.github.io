# 陳睿凱 Mars 個人作品集設計方針
## Portfolio Design Guidelines

---

## 一、核心定位

### 你不是在找工作，你是在讓對的人找到你

這份作品集的根本邏輯：

- ❌ 傳統求職：「我符合這些條件，請錄用我」
- ✅ 這份作品集：「我解決過這些問題，你有類似的問題嗎？」

### 一句話身份定位

> **我找到流程裡的浪費，然後讓它消失。**

這句話取代所有職稱。它說的是你做的事，不是你的頭銜。

---

## 二、頁面結構與「為什麼是我」的位置

### 完整頁面順序

```
1. HEADER（導航）
      ↓
2. HERO（第一印象）
   → 名字 + 身份句 + chips
      ↓
3. 為什麼是我  
   → 轉職故事 + 信念陳述
      ↓
4. 我解決過的問題（案例）
   → 三個展開式案例
      ↓
5. 工具箱（技術）
   → 四個分類，輕量呈現
      ↓
6. 合作 CTA
   → 簡單的邀請與聯繫方式
```

### 為什麼「為什麼是我」放在第三位？

**閱讀心理邏輯：**
- 第一眼（HERO）：「這個人是做什麼的？」→ 建立第一印象
- 第二眼（為什麼是我）：「這個人為什麼這樣？」→ 建立信任與獨特性
- 第三眼（案例）：「這個人做過什麼？」→ 用證據驗證前兩個問題

如果案例放在故事前面，讀者看完技術能力卻不知道你的視角；
故事放在案例前面，讀者帶著理解你思維方式的框架去看案例，說服力倍增。

## 三、各區塊寫作指南

### 區塊 1：HERO

**目標：** 3秒內讓人知道你是誰、你做什麼

**格式：**
```
[名字] · [城市]
[身份句——動詞開頭，不是職稱]
[3-5個能力 chips]
```

**寫作原則：**
- 身份句用動詞開頭，主詞是「我」
- 不用「擁有X年經驗」「熟悉X技術」等 JD 語言
- Chips 描述能力維度，不是職稱

**範例：**
> 陳瑞凱 · 台北
> 我找到流程裡的浪費，然後讓它消失。
> [後端工程] [AI 整合] [流程自動化] [產品思維] [全端開發]

---

### 區塊 2：為什麼是我

**目標：** 建立「只有你才有這個視角」的獨特性

**位置：** HERO 之後，案例之前

**格式：**
```
[一段轉職故事：為什麼走這條非傳統路]
[一句信念陳述：你相信什麼？]
[一個結論：這讓你能做到什麼？]
```

**建議文字（可依自己語氣調整）：**
> 我花了幾年在消費性電子產品的 PM 位置上，然後在某個說不清楚的時刻，決定去學寫程式。不是因為有人建議，而是因為我發現：我永遠比工程師更想知道「這個功能到底在解決誰的什麼問題」，也永遠比 PM 更想親手把解法做出來。
>
> 這個位置讓我很長一段時間沒有辦法填一個整齊的職稱框框。但它也讓我能做到一件很少人能做的事：**從發現問題，到交付解法，全程不需要翻譯。**

**寫作原則：**
- 用第一人稱，語氣真實，不要 HR 語言
- 承認「框框裡放不下你」，把這件事變成優勢
- 結尾要有一個清晰的「所以你能為對方帶來什麼」
- 長度：150-250 字之間，不要太短（顯得沒深度）也不要太長（失去重點）

---

### 區塊 3：案例（我解決過的問題）

**目標：** 用真實的問題解決過程，展示思維方式

**每個案例的固定格式：**

```
[問題標題]：用「問題語言」，不用技術語言
    ↓
展開後顯示：

Before（之前的痛）
→ 用戶/當事人原本在做什麼？花多少時間？有什麼不便？

Think（我怎麼想）
→ 我如何理解這個問題？我的切入點是什麼？
→ 這是展示你思維方式的關鍵欄位，不要只寫技術選擇

After（最後的結果）
→ 具體的改變是什麼？最好有可量化的描述

成果一句話（Result Bar）
→ 用一句話說出最重要的改變

技術標籤
→ 放在最後，對技術感興趣的人才需要看
```

**標題寫法對比：**

| ❌ 技術語言 | ✅ 問題語言 |
|---|---|
| VBA 訂單自動化系統 | 每天手動整理訂單耗費大量時間 |
| AI 影片字幕生成工具 | 影片後製需要手動聽打字幕與整理摘要 |
| Podcast 摘要推播系統 | 想吸收 Podcast 內容但沒時間完整聆聽 |

**Think 欄位寫作原則（最重要）：**
- 不只寫「我用了 X 技術」
- 要寫「我發現問題的本質是 Y，所以我選擇用 Z 方式切入」
- 這個欄位讓你和其他工程師作品集的最大差別：你展示的是判斷力，不只是執行力

---

### 區塊 4：工具箱

**目標：** 讓技術背景清晰，但不讓它成為主角

**呈現原則：**
- 按用途分類，不按重要性排序
- 不列「熟悉度百分比」（進度條很 cliché 且不誠實）
- 不列每一個用過的工具（保留最有代表性的，但現階段因為作品不多可以全部保留）

**建議分類：**
```
後端 / 系統   → Java, Spring Boot, Redis, RabbitMQ, Docker, MySQL
AI / 自動化   → Python, OpenAI API, Whisper, Line API
前端          → React, Vue, Angular
產品 / 流程   → 需求訪談, 流程分析, CI/CD, 敏捷開發
```

---

### 區塊 5：合作 CTA

**目標：** 讓對的人知道怎麼找你

**不要寫：**「歡迎來信洽詢任何合作機會」（太被動，太空泛）

**要寫：**
> 如果你有一個重複在做、但覺得「應該可以更好」的工作流程——我們可以聊聊。

接著提供：Email + LinkedIn（或 GitHub）

## 四、視覺設計原則

### 色彩

| 用途 | 色碼 | 說明 |
|---|---|---|
| 主色（深青綠）| `#0F6E56` | 標題、強調、互動元素 |
| 輔色（琥珀暖色）| `#BA7517` | 點綴、hover 效果 |
| 背景 | `#FAFAF8` | 有機感的米白，非純白 |
| 文字主色 | `#1C1C1A` | 近黑，比純黑更柔和 |
| 文字次色 | `#5F5E5A` | 說明文字、標籤 |
| Before 底色 | `#FCEBEB` | 問題/痛點區塊 |
| Think 底色 | `#EBF3FB` | 思考過程區塊 |
| After 底色 | `#EAF3DE` | 結果/成就區塊 |

### 字體

```css
/* 標題：有個性的襯線體（繁體中文優先）*/
font-family: 'Noto Serif TC', serif;

/* 內文：易讀的無襯線體 */
font-family: 'Noto Sans TC', sans-serif;

/* 程式碼標籤 */
font-family: 'JetBrains Mono', monospace;
```

### 字級與間距

| 元素 | 字級 | 字重 |
|---|---|---|
| 大標題（身份句）| 32-40px | 500 |
| 章節標題 | 18px | 500 |
| 案例標題 | 16px | 500 |
| 內文 | 15px | 400 |
| 標籤/chips | 12px | 400 |
| 行高 | 1.7 | — |

### 排版原則

- **非對稱留白**：不是所有元素都居中，有時刻意靠左創造呼吸感
- **輕重之分**：重要資訊用深色，次要資訊用淺色——讓眼睛有地方落腳
- **邊線使用 0.5px**：細緻、不搶主角
- **圓角**：8-12px，柔和但不幼稚
- **不用進度條、雷達圖表示技能**：這類元素沒有意義

### 互動設計

- 案例卡片：預設收合，點擊展開詳細內容
- Hover 效果：邊線加深（tertiary → secondary），不用陰影
- 展開動畫：簡單的高度過渡，不需要複雜特效
- 手機版：所有卡片改為全寬堆疊

## 五、語氣與聲音

### 你的文字應該讓人感覺

- 真實：不是 LinkedIn 上的公關稿
- 有觀點：不是在描述事實，是在說你怎麼看事情
- 簡潔：每一句都有目的，沒有廢話
- 不自吹：讓問題和結果說話，你只負責描述

### 應該避免的語言

| ❌ 避免 | ✅ 替換 |
|---|---|
| 擅長、精通、熟悉 | 直接說做過什麼 |
| 負責 X 系統的開發 | 解決了 X 問題 |
| 歡迎各種合作機會 | 如果你有這類問題，我們可以聊 |
| 具備跨部門溝通能力 | 用具體案例說明 |
| 成果顯著 | 說出具體的改變是什麼 |

## 六、未來更新節奏

### 作品集是活的文件，不是一次性的

| 時間點 | 更新內容 |
|---|---|
| 工作室案子完成後 | 加入第四個案例（第一個付費客戶案例）|
| 每完成一個新專案 | 評估是否比現有案例更有代表性 |
| 每半年 | 重新審視身份句是否仍然準確 |
| 收到客戶/面試官反饋 | 調整描述語言以符合對方真實關心的點 |

### 案例優先順序原則

新的案例要進入作品集的標準：
1. 解決的問題比現有案例更有代表性
2. 有更具體可量化的結果
3. 展示了新的技術能力維度
4. 是付費客戶（優先度最高）

## 七、快速檢查清單

發布前確認：

- [ ] 身份句用動詞開頭，不是職稱
- [ ] 「為什麼是我」有個人故事，不是能力清單
- [ ] 每個案例標題用問題語言，不是技術語言
- [ ] Think 欄位寫的是思維過程，不只是技術選擇
- [ ] 技術標籤在案例的最後面
- [ ] 沒有「擅長」「精通」「熟悉」等空洞詞彙
- [ ] 手機版確認排版正常
- [ ] CTA 說清楚什麼樣的人應該找你
