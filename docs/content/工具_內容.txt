# 庫存表自動生成工具 AUI

before: "每週固定從 ERP 匯出三份格式各異的報表，再逐欄手動比對八個倉別數字，整個流程穩定耗費一小時，還擔心哪個數字算錯、倉別搞混。"
think: "問題的本質是格式轉換加欄位映射——重複且有規則可以被程式化。選 Python 而非 Excel，是因為它可以進 Git 版本控制、打包成 exe 讓同事不需開發環境即可執行。"
after: "一行指令，五分鐘後拿到格式正確、欄位齊全的彙整表，可以直接帶進會議室。Python 打包成 exe，同事不需安裝任何工具就能執行。"

## 問題背景

在亞果元素擔任 PM 期間，每週固定要整理一份庫存表在部門週會上使用。這件事聽起來簡單，做起來卻很折磨：先從 ERP 匯出三份報表（存貨彙總、銷售訂單、客戶出貨與退貨），每份格式都不同；再逐欄手動比對不同倉別的數字——T+M 倉、官網電商倉、門市倉、工廠倉、寄倉量……一共八個倉別，不只需要手動調整新品，如果倉別有新增的話，也需要手動修改excel的格式。

整個流程從匯出到整理完成，穩定要花一個小時，而且每次到最後都提心吊膽，怕哪個數字算錯、搞混哪個倉別。這不是一次性的任務，而是每週都要重複的負擔。

## 解決方案

我寫了一支 Python 程式，讓這件事變成一行指令：

```shell
aui.exe 20991230 windows
```

把三份 ERP 報表放進指定資料夾，執行指令，五分鐘後拿到格式正確、欄位齊全的 `ADAM產品庫存_20991230_for_Meeting.xlsx`，可以直接帶進會議室。程式打包成 `.exe`，連 Python 都不用安裝，同事也能直接執行。

## 技術決策

選 Python 而不是 Excel VBA，主要有三個原因：一、Python 可以放進 Git 做版本控制，欄位規則怎麼改都有記錄；二、打包成 `.exe` 讓非工程師同事不需要任何開發環境就能執行；三、日後如果公司需要把這個流程排進 CI 自動跑，Python 比 VBA 更容易整合。

倉別的欄位映射規則統一放在設定檔 `warehouse_codes.xlsx` 裡維護，不寫死在程式碼中。這樣以後倉別增減，只需改設定檔，不需要動程式邏輯。

## 挑戰與踩坑

最大的麻煩是不同 ERP 報表的欄位命名和日期格式不統一。同樣是「出貨日期」，不同報表可能叫不同名字，日期格式也可能是 `20991230` 或 `2099/12/30` 或 `2099-12-30`。如果程式直接信任輸入，遇到格式變動就會靜默算錯，比手動整理更危險。

解法是定義一套欄位映射規則加上格式驗證：程式在讀取每份報表時，先比對預期欄位清單，若發現不符合就立刻報錯並指出是哪份來源，強迫使用者確認而不是讓錯誤悄悄流入輸出。

另一個挑戰是整合所有報表的邏輯，其中有太多手動調整的地方，不一定有明顯的規則，都是透過經驗累積出來，並且利用這次機會整理出對應的邏輯，透過程式碼的規則化，可以更容易的維護和擴展。

# 年節訂單報表彙整

before: "客人電話訂年菜，服務生手寫三聯單，行政再逐筆 key 進 POS，POS 資料再謄進 Excel——三道工序全程手工，年節旺季整理報表每天都要花半小時到一小時才能完成。"
think: "整個流程其實只有一個真正的資料來源：POS 匯出的 CSV。多餘的步驟來自「資料格式不對，人要去補洞」。用 VBA 直接讀取 CSV 並清洗格式，把三道工序壓成一個按鈕。"
after: "把 CSV 放進資料夾，按一個按鈕，統計表自動完成。格式統一、幾乎零錯誤，人力從整理資料移到使用資料。"

## 問題背景

親戚在餐飲業做行政，每逢農曆年節前都要面對同一個惡夢：客人打電話訂年菜，服務生手寫三聯單；行政再把三聯單一筆筆 key 進 POS 機；POS 資料進去之後，還要再從三聯單逐筆謄到 Excel，才能統計各品項訂購量。

三道工序、全程手工，不只耗時，更容易漏單。年節旺季訂單量一大，光是整理報表就要花掉半個工作天，而且每次都擔心哪筆數字搞錯、哪個客人的訂單沒對上。

## 解決方案

我幫她寫了一支 Excel VBA 巨集，讓流程縮減成三個步驟：

1. 將 POS 匯出的 `訂貨明細表.csv` 放進同一個資料夾
2. 開啟 `115年度_總單.xlsm`，執行巨集 `ImportAndCleanPOSCsv`
3. 更新樞紐分析表，統計表格就完成了

不需要再從三聯單二次謄寫，POS 裡的資料就是最終報表的來源。巨集同時支援多個 CSV 一次匯入合併，整個年節的訂單可以一口氣處理完。

## 技術決策

這個案子選 Excel VBA 而不是獨立程式，不是技術妥協，而是務實的選擇。餐飲業環境不需要另外安裝任何軟體，所有人電腦上都有 Excel，學習成本幾乎是零。對這個規模的需求，VBA 的能力完全足夠，多餘的工程複雜度反而會造成維護困難。

巨集還整合了一份 `品項對應表.xlsx`，把 POS 系統的原始品項名稱對應到報表用的簡化名稱——例如「年菜外帶4988」統一轉成「年菜外帶」。對應規則存在外部檔案裡，餐飲業自己就能維護，不需要工程師介入。

## 挑戰與踩坑

最棘手的問題有兩個。第一，POS 機匯出的 CSV 欄位格式在不同日期之間有微小差異，同一個欄位的命名偶爾會變動。解法是加入欄位自動偵測邏輯，程式不依賴固定欄位位置，而是依名稱動態找到對應欄，確保即使欄位順序改變也能正確解析。

第二，POS 系統把手機號碼存成數字格式，匯出 CSV 之後前導零會自動消失，`0912345678` 變成 `912345678`。巨集在匯入時會自動補回前導零，讓聯絡資料保持正確格式，客人電話打得通。

最後一個挑戰是資料的準確性，因為原始資料是從 POS 機匯出，如果員工因為忙碌或者疏忽沒有同步更新到 POS 系統，則會導致資料不準確。

快速用藥查詢工具

before: "診所快速用藥清單是全院共用的，自己常開的處方混在所有醫師的設定裡，每次查詢都要翻半天，在看診節奏快的環境下每天累積成可觀的時間壓力。"
think: "問題不是系統不夠好，而是使用者需要「只有自己邏輯」的個人工具。選 Google Apps Script + Sheets：讓醫師自己維護資料、不依賴工程師，也不需要伺服器費用。"
after: "輸入關鍵字立刻找到對應藥品，支援中英文名、成分、代號四種搜尋方式。手機電腦都能開，不需安裝任何程式。"

## 問題背景

朋友在診所擔任醫師，診間裡雖然有醫療系統的快速用藥組合功能，但有個麻煩：那份清單是全院共用的，每個醫師的用藥習慣都混在一起，找自己常開的處方反而要翻半天。她想要一個個人專屬的查詢介面，只有她自己的用藥邏輯，不會被其他同事的設定干擾。

看診節奏快，每位病患的時間有限。如果查個藥名或確認一下用藥組合要花兩分鐘，一個上午十幾位病患累積下來就是不小的時間壓力。

## 解決方案

我用 Google Apps Script 搭配 Google Sheets，做了一個可以用瀏覽器開啟的個人查詢工具。介面簡潔：輸入關鍵字就能搜尋藥品（支援中文名、英文名、成分、代號四種方式），也可以依症狀或病症大類篩選她常用的用藥組合。

藥品資料存在 Google Sheets 裡，包含代號、藥品名稱、英文名稱、成分、劑型和健保價。她隨時可以直接在 Sheets 裡新增或修改資料，不需要找我幫忙。查詢工具透過 Web App URL 開啟，手機或電腦都能用，也不需要安裝任何程式。

## 技術決策

選擇 GAS + Google Sheets 而不是獨立後端服務，主要考量是零摩擦部署。透過 Google Web App URL 直接存取，不需要伺服器費用、不需要維護機器；資料存在 Sheets，醫師本人就能獨立維護，不依賴工程師支援。

藥品資料庫特別加入了健保價欄位，這是有意識的設計——讓醫師在選藥時可以快速參考費用，輔助臨床決策，而不只是查藥名而已。

## 挑戰與踩坑

這套工具目前是原型狀態，尚未在真實看診環境正式部署。醫療工具的開發有一道額外的門檻：不能讓使用者對工具的可靠性產生過度信任。

因此介面上明確加入免責說明，提醒這是個人查詢輔助工具，最終用藥決策仍由醫師判斷，工具只是幫忙加快找資料的速度，而不是取代臨床判斷。這個邊界在原型階段就要設計清楚，之後才方便在真實環境中安全部署。

股市資產監控儀表板

before: "多家券商帳戶分開登入、逐筆記下持倉、手動加總換算台幣——每次想了解整體損益都要花半小時，而且算完仍然看不出哪些資金在有效工作。"
think: "核心問題是「資料分散」加上「缺少比較視角」。把所有帳戶交易記錄整合進同一份 Sheets，前端用持倉熱度排行視覺化漲跌動能，讓「哪支股票的錢在睡覺」一眼可見。"
after: "開啟 Web App，自動讀取記錄、抓取即時股價、計算 FIFO 損益，兩分鐘內看完整體狀況，可依持有人篩選不同帳戶。"

## 問題背景

我自己透過幾家不同的券商購買美股，多個帳戶同時持有多支標的。每次想了解整體損益，流程都一樣麻煩：先登入 A 券商看持倉，記下數字；再登入 B 券商再記一次；最後打開計算機手動加總，換算成台幣再做比較。

這個過程穩定要花半小時，而且每次算完都不確定「資金配置是否合理」——有些股票長期抱著，漲幅很小，資金就這樣靜靜躺著。沒有一個地方可以快速看出哪支股票的錢在有效工作、哪支的錢在睡覺。

## 解決方案

我用 Google Apps Script 做了一個整合儀表板，所有交易記錄統一存在一份 Google Sheets 裡。開啟 Web App，系統自動讀取記錄、抓取即時股價、計算 FIFO 損益，兩分鐘內就能看完整體狀況。

儀表板的核心功能是「持倉熱度排行」：依近 30 天漲幅把持倉分成四個等級（強勢、上漲、盤整、下跌），一眼看出哪些標的在積極貢獻報酬、哪些資金長期沒有動靜。KPI 卡片顯示總投入成本、當前市值、總報酬和活躍持倉數，也可以依持有人篩選，區分不同人的帳戶。

## 技術決策

選 GAS + Google Sheets 的原因和技術門檻都不高：交易記錄本來就適合試算表格式，GAS 直接讀取計算，零部署成本，不需要另外架伺服器。前端用 Chart.js 4.4.0 呈現視覺化，在 GAS 的 Web App 架構下可以直接嵌入 HTML 使用。

成本計算採用 FIFO 先進先出而非平均成本，這是有意識的選擇。分批買入同一支股票時，FIFO 能更精確反映「這批買入的實際損益是多少」，對資金管理判斷更有參考意義。幣別以 USD 為主，透過 exchangerate-api.com 即時換算 TWD，不需要手動更新匯率。

## 挑戰與踩坑

最主要的限制是股價延遲問題。Google Sheets 內建的 `GOOGLEFINANCE()` 函式有 15 到 20 分鐘的延遲，沒辦法做到真正即時。

面對這個限制，解法是明確定位工具的使用場景：這個儀表板是「持倉參考」工具，目的是了解整體配置狀況，而不是用來做即時交易決策。介面上直接標示延遲說明，讓使用者清楚知道數字的時效性。這個取捨在這個使用場景下完全合理——我需要的是「整體有沒有賺、哪個方向對」，而不是精確到秒的價格。

影片轉錄與摘要工具

before: "課程影片和 Podcast 沒有任何文字整理，想複習特定知識點只能重新播放影片。Whisper 的中文輸出以簡體為主，對繁體中文使用者閱讀體驗很差，手動整理逐字稿又太耗時。"
think: "謄稿是「人做沒有價值、機器做完全可以」的典型工作。選 faster-whisper（比原版快四倍）加 OpenCC 後處理，讓本地執行直接輸出繁體中文，不需 API 費用，也保護課程影片的資料隱私。"
after: "把影片拖進工具，自動輸出 .srt 字幕檔與帶時間戳記的 Markdown 筆記。macOS 版雙擊開啟，不需要終端機。"

## 問題背景

我需要持續上大量美容醫學課程進修，但課程只有影片錄影，沒有任何文字整理。每次想複習某個知識點，只能重新播放影片，在腦中重建內容。事後複習效率極低，遇到考試或需要查詢特定資訊時更是麻煩。

後來我自己也開始用，主要用來處理 Podcast。很多節目品質很好，但我沒時間完整聽完一集，卻又想快速掌握重點。手動整理逐字稿太耗時，又沒有符合需求的工具可以直接輸出繁體中文的結構化筆記。後來因為需要處理醫美課程的影片，所以開發了這個工具。

## 解決方案

我開發了一套本地執行的影片轉錄工具，支援 CLI 和 GUI 兩種操作方式。把影片檔案拖進去，工具自動完成：用 FFmpeg 提取音訊、用 faster-whisper 在本機跑語音辨識、用 OpenCC 把辨識結果從簡體轉換成繁體中文。

輸出兩份檔案：`.srt` 字幕檔可以直接掛回影片播放器，`_notes.md` 是帶時間戳記的結構化 Markdown 筆記，可以直接在任何 Markdown 編輯器裡閱讀或搜尋。macOS 版有包裝成 `.app`，雙擊就能開啟 GUI，不需要開終端機。

```bash
# 單一影片
python -m video_notes lecture.mp4

# 批次處理整個課程資料夾
python -m video_notes ./lectures/
```

## 技術決策

核心決策是選用 `faster-whisper` 而不是 OpenAI 的原版 Whisper，理由很直接：same 模型權重、4 倍速度提升、中文辨識準確率相近。更重要的是完全本地執行，不需要 API 金鑰、不需要網路連線、沒有 API 費用，也不需要把語音資料傳到外部伺服器——對醫療相關課程來說，資料隱私是不能妥協的考量。

加入 OpenCC 這一層轉換是針對繁體中文使用者的明確設計。Whisper 的中文輸出以簡體為主，對台灣使用者來說閱讀體驗很差。OpenCC 後處理讓輸出直接是可讀的繁體中文，不需要使用者再手動轉換。

## 挑戰與踩坑

遇到兩個主要問題。第一，長影片（超過一小時）在本機跑完整轉錄需要很長時間，使用體驗很差。解法是切分成多段批次處理再合併，讓使用者可以中途看到進度，也方便在較低規格的硬體上執行。

第二，醫美領域的術語辨識準確率偏低——藥物學名、儀器品牌名這類專業詞彙，語音辨識模型經常辨識錯誤。這個問題目前採人工校對的方式處理，還沒有做自訂詞彙表的優化。這是已知的限制，在工具說明裡有明確標示，使用者使用前就知道這類內容需要校對。
